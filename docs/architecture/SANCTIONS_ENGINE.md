## 0. Цель

Единый **централизованный движок «действие → противодействие»** на всех уровнях: автономно сервером (флуд, перебор, спуф, стоп-слова, рецидив) и вручную модераторами/админами/овнерами. Заменяет россыпь разрозненных проверок и хелперов одним инструментом. Даёт полный аудит и статистику.

Принцип: **новое прирастает конфигом и детекторами, а не переписыванием кода.**

## 2. Архитектура: единая точка `SanctionService`

Все источники проходят через один сервис. Это и есть устранение костылей.

```
ИСТОЧНИК                                  SanctionService.apply(intent)
 • модератор/админ/овнер (WS room_action) ─┐  1. AccessContext: политика + инварианты (I-1..I-15)
 • админ-панель (HTTP) ────────────────────┤  2. validate: тип/срок/причина (I-4/I-5)
 • АВТОНОМНЫЙ актор (детекторы) ────────────┤  3. escalate: рецидив по типу → ступень лестницы
                                           │  4. circuit-breaker: анти-аномалия (см. §9)
                                           │  5. write: moderation_events + active_restrictions (1 транзакция)
                                           │  6. effects: сессия/force_logout/WS (I-9/I-10)
                                           └► 7. emit: WS-событие (через мост HTTP→WS если источник HTTP)

SanctionService.lift(intent)  — снятие: те же инварианты (I-6/I-8), запись unban-события с parent_event_id.
```

**Заменяет:** `RoomController::mute/ban`, `UserManager::ban/roomUnban/roomUnmute`, прямые проверки `is_banned`/`muted_until`/`room_role='banned'`. После миграции горячие проверки идут по `active_restrictions`.

## 3. Типы санкций и кто их выдаёт

| Санкция | Кто выдаёт | Эффект | Срок |
|---|---|---|---|
| **kick** | люди (mod/admin/owner) | разрыв сессии+WS, без долгого запрета | — |
| **mute** | **ТОЛЬКО люди** (mod/admin) | запрет писать в комнате, сессия живёт | по политике |
| **ban_room** | люди + **система** | запрет входа в комнату, сессия рвётся | лестница |
| **ban_global** | люди (admin+) + **система** | запрет входа в систему, сессия рвётся | лестница |

**Автобан = только система, и только баны (не мьюты).** Мьют — человеческий инструмент.

## 4. Лестница автобана и вход по коэффициенту

Ступени: **1ч → 3ч → 24ч → 7д → 30д → permanent.**

**Точка входа определяется коэффициентом опасности нарушения:**

| Класс нарушения | Коэффициент | Стартовая ступень | Климб по рецидиву |
|---|---|---|---|
| Слова / спам | низкий | **1ч** | 1ч → 24ч → 7д → 30д → permanent (3ч пропускается) |
| Атаки (brute-force, спуф) | высокий | **минимум 3ч** | 3ч → 24ч → 7д → 30д → permanent |
| Критичные (подтв. XSS, захват сессии) | максимум | **прыжок вверх / permanent** | сразу по коэффициенту |

**Эскалация по рецидиву:** счётчик ведётся **раздельно по типу нарушения**. Набралось N нарушений типа T на текущей ступени → следующая ступень. Пороги N — в конфиге (старт: стоп-слово = 5, brute-force = 10/15мин).

**`permanent` (system)** снимает **только owner** (platform_owner), и только при достижении через высокий коэффициент. Обычные авто-санкции (≤30д) снимаются по обычным правилам «строго выше».

---

## 5. Коэффициент опасности

Веса по типам (стартовая калибровка, крутится в конфиге, тюнится в shadow):

| Нарушение | Вес |
|---|---|
| Атака на аккаунт/сессию (спуф, кража токена, масс-регистрация) | 10 |
| Вредоносный контент (XSS/скрипт, фишинг) | 8 |
| Флуд/DoS (шквал сообщений/реконнектов) | 7 |
| Стоп-слова / хейт | 4 |
| Спам (повторы, ссылки, mention-бомбинг `@!`) | 3 |
| Разовый срыв rate-limit | 1 |

**Формула:** `coefficient = Σ(вес × свежесть) × множитель_рецидива`
- свежесть: <24ч полный вес, <7д ½, <30д ¼;
- множитель рецидива: `1 + 0.5 × число_санкций_за_30д`.

**Коэффициент → выбор стартовой ступени и скорости климба.** Мгновенные триггеры (вес ≥8, подтверждённые) могут идти сразу в permanent.

> Веса и пороги — **стартовая гипотеза**, не истина. Калибруются в shadow-режиме (§9) на реальных событиях до боевого включения.

## 6. Автономные детекторы

Каждый детектор при срабатывании зовёт `SanctionService.apply` с `origin='system'`, `actor_role='system'`, `trigger_code=<тип>`.

| Детектор | Триггер | trigger_code | Старт |
|---|---|---|---|
| **Brute-force** | 10 неудач логина / 15 мин (на аккаунт И на IP) | `bruteforce` | 3ч |
| **Флуд** | превышение rate-limit N раз подряд | `flood` | 3ч |
| **Стоп-слова** | совпадение со списком (см. §7) | `stopword` | 1ч |
| **Спуф/аномалия сессии** | рассинхрон/подмена | `spoof` | 3ч+ |
| **Рецидив** | накопленные санкции по типу | `recidivism` | следующая ступень |

### 6.1 Защита от brute-force (детально)

Закрываем оба вектора:

| Вектор | Атака | Счётчик |
|---|---|---|
| Targeted | 1 аккаунт, много паролей | по `account_id` |
| Spray | много аккаунтов, 1 пароль, 1 IP | по `ip` |

Механизм (сервер, счётчики в БД — PHP-FPM stateless):
1. Таблица `login_attempts(key, fail_count, window_start)` — key = account_id или ip.
2. Прогрессивная задержка ответа после 5 неудач.
3. 10 неудач / 15 мин → системный автобан 3ч на логин (`trigger_code='bruteforce'`, событие в лог).
4. Generic-ошибка «неверный логин или пароль» — не раскрывать, что именно неверно (анти-энумерация). Проверить `LoginHandler`.
5. `password_verify` — constant-time (уже есть).
6. Опционально позже: CAPTCHA-челлендж после порога.

## 8. IP: захват, статистика, «волк в овечьей шкуре»

### 8.1 Захват
- `+actor_ip`, `+target_ip` в `moderation_events` — снимок в момент инцидента (как `actor_role`). **Не дубль:** сессии удаляются при бане, поэтому IP фиксируется в вечном логе. Отдельная IP-таблица не нужна — статистика это запросы к событиям.
- Ретеншн: события не стираются (решение владельца). Privacy-нюанс зафиксирован осознанно.

### 8.2 IP-бан — отложен
Блокировка по IP — позже и осторожно (коллатераль: CGNAT/общий Wi-Fi бьёт по невиновным). Сейчас — только захват для статистики и интел-сигнала.

### 8.3 Интел-сигнал «волк в овечьей шкуре»
**Нарушение санкционируется ВСЕГДА** (аккаунт-нарушитель банится по своим правилам — нарушение не пропускаем никогда). **Дополнительно** — алерт стаффу для детального разбора:
1. Индекс **IP → аккаунты** из событий + активных сессий.
2. Правило: за одним IP ≥2 аккаунта, ≥1 под санкцией, другой активен → алерт «возможный обход/мульти-акк: IP X → [забанен A, активен B]».
3. Вторичные сигналы: общий User-Agent, всплеск регистраций с IP.
4. Куда: панель «Алерты/Интел» для admin+owner. **Авто-бан второго аккаунта только по факту общего IP — НЕ делаем** (коллатераль); решает человек.

## 9. Safety-рельсы (критично)

Автоном без тормоза опасен. Закладываем сразу:

1. **Иммунитет стаффа:** `system` проходит инварианты I-1/I-3 — не трогает platform_owner/admin/moderator.
2. **Авторитет `system`:** уровень admin (5) + право на `permanent` (снимает только owner).
3. **Circuit-breaker:** не более N авто-санкций/мин по платформе (старт ~10). Превышено → автоном **на паузу**, алерт owner, ручное включение (fail-safe).
4. **Нет повторной санкции:** `active_restrictions` UNIQUE + проверка перед применением — петля невозможна.
5. **Нет рекурсии:** события `origin='system'` сами не служат триггером.
6. **Kill-switch:** owner выключает автоном одним флагом (`app_settings`).
7. **Shadow-режим по умолчанию:** детектит и пишет в лог `would_sanction`, но **не банит**, пока owner не переведёт в боевой. Калибровка весов/порогов на реальных данных.

## 10. Авточитаемая причина

Система формирует человеко-понятную причину с объяснением эскалации, не «system»:

> `Автобан 24ч: 5× нарушение «стоп-слово», предыдущие санкции 1ч×4. Эскалация по рецидиву.`

Шаблон: `<санкция> <срок>: <N>× <тип>, предыдущие <история>. <механизм>.`

## 11. Контракт событий и доставка

- WS-события: `force_logout` (kick/ban, I-9), `muted_in_room`/`unmuted_in_room`, `banned_*`. Единый реестр, валидируемый.
- **Мост HTTP→WS:** действия из админ-панели (HTTP/PHP-FPM) должны доходить до клиентов. Outbox-таблица: HTTP пишет событие → WS-процесс поллит (~1с) → broadcast. Localhost, без нового открытого порта. Это закрывает «снятие из админки требует F5».
- Клиент: единая точка приёма событий → реактивное отражение состояния (см. отдельный фронтенд-механизм).

## 12. Статистика

Дашборды = запросы к `moderation_events` (индексы уже есть):
- по датам (`idx_act_time`);
- по IP (`actor_ip`/`target_ip`) — кто чаще атакует;
- по модераторам (`idx_actor`) — кто чаще банит;
- по типам атак (`trigger_code` через **функцию-классификатор**, не жёсткий ENUM — открытый словарь, новый тип без миграции).

## 15. Конфиг по умолчанию (калибруется в shadow)

```
escalation:
  stopword:   { start: 1h, threshold: 5,  ladder: [1h,24h,7d,30d,permanent] }
  bruteforce: { start: 3h, window: 15m, attempts: 10, ladder: [3h,24h,7d,30d,permanent] }

danger_weights: { auth_attack:10, malicious_content:8, flood:7, stopword:4, spam:3, ratelimit:1 }
danger_thresholds: { '1h':5, '3h':15, '24h':30, '7d':50, '30d':65, permanent:80 }
circuit_breaker: { max_system_sanctions_per_min: 10 }
mode: shadow   # shadow | live
```
